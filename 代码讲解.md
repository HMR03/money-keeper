# 记账本 PWA — 代码讲解（面向初学者）

这份文档逐部分讲解项目的代码结构和设计思路，适合想学习「纯前端 PWA 是怎么做出来的」的读者。

---

## 一、项目是什么

这是一个**纯前端**的记账应用，可以像原生 App 一样安装到手机主屏幕，断网也能用。

关键词：

| 概念 | 大白话解释 |
|------|-----------|
| PWA | Progressive Web App，网页穿上 App 的壳，能装到桌面、离线打开 |
| SPA | Single Page Application，只有一个 HTML 文件，靠 JS 切换页面 |
| IndexedDB | 浏览器内置的数据库，能存大量结构化数据 |
| localStorage | 浏览器内置的键值对存储，适合存少量配置 |
| OPFS | Origin Private File System，浏览器给每个网站分配的私密文件夹 |
| Service Worker | 网页的后台线程，能拦截网络请求，实现缓存和离线 |

---

## 二、文件结构

```
记账本PWA/
├── index.html          ← 整个 App 的唯一页面（HTML + CSS + JS 全在这）
├── manifest.json       ← PWA 配置文件（图标、名称、全屏模式等）
├── sw.js               ← Service Worker（离线缓存逻辑）
├── icons/
│   ├── icon-192.png    ← App 小图标
│   └── icon-512.png    ← App 大图标
├── 记账本完整使用指南.txt  ← 用户使用说明
└── README.md           ← 项目说明
```

**为什么只有一个 HTML 文件？**

因为项目足够小，把所有东西放一个文件里反而简单——不用装 Node.js、不用配 Webpack，浏览器直接打开就能跑。坏处是代码长了不好维护，适合小型项目。

## 三、index.html 逐段讲解

### 3.1 头部 `<head>`（第 1-11 行）

```html
<meta name="viewport" content="width=device-width, initial-scale=1, maximum-scale=1, user-scalable=no, viewport-fit=cover">
```

- `viewport-fit=cover` → iPhone 刘海屏全屏显示
- `user-scalable=no` → 禁止用户双指缩放，更像原生 App

```html
<meta name="apple-mobile-web-app-capable" content="yes">
```

- 告诉 iOS Safari：这个网页可以全屏运行（隐藏地址栏）

```html
<meta name="theme-color" content="#FF6B35">
```

- 浏览器地址栏/状态栏的颜色，和 App 主题色保持一致

```html
<link rel="manifest" href="manifest.json">
```

- 指向 PWA 清单文件，浏览器靠这个知道怎么「安装」这个 App

```css
:root {
```

```javascript
function numInput(key) {
  if (key === 'del') {
    // 删除最后一个字符，剩一个时变回 '0'
    addAmount = addAmount.length > 1 ? addAmount.slice(0, -1) : '0';
  } else if (key === '.') {
    // 只能有一个小数点
    if (!addAmount.includes('.')) addAmount += '.';
  } else {
    // 避免前导零
    if (addAmount === '0') addAmount = key;
    else addAmount += key;
  }
  // 限制小数两位、总长 10 位
}
```

## 四、JavaScript 核心逻辑

### 4.1 数据存储 —— IndexedDB（第 319-371 行）

IndexedDB 是浏览器里的数据库，比 localStorage 能存更多数据，支持索引查询。

```
数据库名：MoneyKeeper（版本 1）
  └── 表：transactions
        ├── 主键：id（自动生成）
        └── 索引：yearMonth（按月查询）
```

一条记账记录的结构：

```javascript
{
  id: "abc123",           // 唯一 ID（时间戳 + 随机数）
  type: "expense",        // "expense" 支出 或 "income" 收入
  amount: 35.5,           // 金额
  category: "餐饮",        // 分类
  note: "午餐",           // 备注（可选）
  date: "2026-06-21",     // 日期
  yearMonth: "2026-06",   // 年月（用来按月查询）
  createdAt: 1718960000   // 创建时间戳
}
```

**核心数据库操作函数**：

| 函数 | 作用 |
|------|------|
| `openDB()` | 打开/创建数据库 |
| `dbAdd(tx)` | 新增或更新一条记录（put 是 upsert） |
| `dbDelete(id)` | 删除一条记录 |
| `dbGetByMonth(year, month)` | 按月查询（走索引，很快） |
| `dbGetAll()` | 读取全部记录 |
| `dbClear()` | 清空数据库 |

### 4.3 图片存储 —— OPFS（第 894-920 行）

OPFS 是浏览器给每个网站分配的私密文件夹，适合存文件。

```
图片流程：
拍照/选图 → compressImage() → Canvas 缩到 800px 宽 → JPEG 0.7 质量
           → saveImageToOPFS(txId, blob) → 存为 img_{txId}.jpg
```

**为什么压缩？**
- 手机照片通常 3-5MB，存原图浪费空间
- 压缩后约 60KB，记账凭证级别的清晰度完全够
- 压缩在 Canvas 里完成，不经过服务器

**图片生命周期**：

| 操作 | 发生的事 |
|------|---------|
| 新建 + 拍照 | 压缩 → `saveImageToOPFS()` |
| 编辑 + 换图 | 同名文件覆盖旧图 |
| 编辑 + 点「✕ 移除」 | `pendingImage = 'REMOVED'` → 保存时调 `deleteImageFromOPFS()` |
| 删除整条记录 | `dbDelete()` → `deleteImageFromOPFS()` |
| 不碰图片 | 什么都不发生，零影响 |

## 五、PWA 工作原理

### 5.1 manifest.json

```json
{
  "name": "记账本",
  "display": "standalone",    // 全屏，隐藏浏览器地址栏
  "theme_color": "#FF6B35",   // 状态栏颜色
  "orientation": "portrait",  // 锁定竖屏
  "icons": [...]
}
```

浏览器读到这个文件就知道：这是一个可以安装的 App，用这些图标，全屏运行。

### 5.2 Service Worker（sw.js）

```
用户第一次打开网页
  ↓
浏览器下载 sw.js
  ↓
SW 的 install 事件触发
  ↓
预缓存 index.html、manifest.json、图标、Chart.js CDN 到 CACHE
  ↓
下次用户打开（断网也能用）
  ↓
SW 的 fetch 事件拦截请求
  ↓
先从缓存找 → 找到就用缓存的（秒开）
            → 找不到才联网 → 联网成功存缓存
                            → 联网失败返回 index.html（离线兜底）
```

**为什么改代码后手机不更新？**

SW 使用 cache-first 策略：缓存有就直接用，不联网检查。

解决方法：每次改完代码，把 `sw.js` 第一行的 `CACHE_NAME` 版本号 +1：

```javascript
const CACHE_NAME = 'moneykeeper-v4'; // ← 改代码后 +1
```

浏览器检测到 sw.js 变了 → 安装新 SW → 新 SW 缓存最新文件 → 激活后删掉旧缓存。

### 5.3 OPFS 静默备份

```javascript
// 每次保存后偷偷多存一份
async function backupToOPFS() {
  const root = await navigator.storage.getDirectory();
  const file = await root.getFileHandle('moneykeeper_backup.json', { create: true });
  // 把全量数据写入
}

// 启动时检测：数据丢了？从备份恢复
(async function init() {
  const all = await dbGetAll();
  if (!all.length) {
    await restoreFromOPFS();  // 尝试恢复
  }
})();
```

**为什么需要这个？** IndexedDB 在极端情况下可能被浏览器清理（比如存储空间不足）。OPFS 文件更稳定，系统清理时不删。

## 七、关键设计决策

| 决策 | 为什么 |
|------|--------|
| 单文件架构 | 项目小，一个 HTML 够用；初学者容易看懂 |
| IndexedDB 而不是 localStorage | 交易记录可能成千上万条，需要索引查询 |
| 图片存 OPFS 不存 IndexedDB | 二进制太大，污染数据库查询性能 |
| 数字键盘自建 | 手机上 `<input type="number">` 体验差，自建键盘可控 |
| Chart.js CDN | 纯报表页用，不安装 npm 包 |
| 纯前端无后端 | 隐私第一，数据都在用户自己手机上 |
| CSS 变量 | 统一配色，改一个变量全局生效 |

## 八、怎么自己改代码

- **换颜色**：改 `:root` 里的 `--primary` 变量，整个 App 的橙色全部变成你选的颜色
- **加分类**：改 `DEFAULT_EXPENSE_CATS` 或 `DEFAULT_INCOME_CATS` 数组
- **改图标**：换 `icons/` 目录下的 PNG 文件
- **调布局**：CSS 在 `<style>` 标签里，按页面分组有注释
- **加功能**：HTML 加结构 → CSS 加样式 → JS 加逻辑，按现有模式来

**改完代码别忘了**：`sw.js` 里 `CACHE_NAME` 版本号 +1，否则手机看不到更新。
